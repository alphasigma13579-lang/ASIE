# الخطة الرئيسية لجاهزية منتج ASIE

## 1. التفويض

تحول هذه الخطة ASIE من محرك تحليل محلي تم التحقق من مساره إلى منتج SaaS موثوق وقابل للتسويق. وهي تحل محل ترتيب الحزم `PF-02` إلى `PF-07` في خطة ما بعد التجميد، بينما تبقى `PF-00` و`PF-01` صحيحتين كعمل تأسيسي لخط الإصدار.

هدف المنتج ليس نسخة أولية تقنية نحيفة. الهدف هو **الحد الأدنى من المنتج الجدير بالثقة**: رحلة عميل كاملة، آمنة ومفهومة وذات حضور بصري رفيع، يرى فيها العميل قيمة واضحة قبل أن يطلب قدرات متقدمة.

## 2. أطروحة المنتج

يجب أن يسوق ASIE لنفسه عبر كل نقطة تفاعل، من دون إخفاء الحقيقة أو اختلاقها.

- التجربة العامة تشرح قيمة القرار بوضوح وتستحق اهتمام العميل.
- مساحة العميل تحول المدخلات والأدلة المحلية إلى رحلة قرار واثقة.
- مركز القيادة يعرض القرار الأحدث ومؤشرات الأداء والمخاطر والخطوة التالية فوراً.
- طبقة إدارة المنصة تمكّن المشغل من إدارة العملاء والوصول وصحة الخدمة والاشتراكات والدعم بأمان.
- كل قيمة محكومة قابلة للتتبع إلى Snapshot غير قابل للتغيير.

هدف التصميم هو مرصد قرارات عربي أولا: كثيف تشغيلياً، متزن بصرياً، وهادئ. يجوز استخدام إخراج بصري تقني أصلي، وطبقات للبيانات، وحركة رصينة، وتسلسل هرمي قوي. لا يجوز تقليد أعمال فنية تاريخية، أو تقديم بيانات زخرفية على أنها حقيقة، أو تحويل أداة العمل بعد تسجيل الدخول إلى صفحة تسويقية.

## 3. خط الأساس الحالي

- `AAS Runtime Freeze v1.0` فعال منذ `2026-07-19T00:22:00+03:00` في `Asia/Riyadh`.
- مسار Project Run والمخرجات المختومة وSnapshot Assembly وإسقاطات Snapshot مجمدة.
- آخر تحقق كامل مرجعي: 130 اختباراً، وPython compile، وبناء الواجهة مر بنجاح.
- لا توجد شبكة خارجية مفعلة، ولا مفاتيح، ولا API حكومي، ولا مزود دفع، ولا مزود AI.
- المنتج الحالي يدعم المشاريع والأدلة والتشغيل وSnapshot والتقرير والقرار.
- المنتج الحالي لا يملك بعد نطاقاً حقيقياً للهوية أو المنظمات أو الأدوار أو الفوترة أو المراسلات أو لوحة التحكم التشغيلية.

## 4. بوابة الوصول الخارجي غير القابلة للتفاوض

لا يجوز تفعيل تسجيل عام، أو تكامل خارجي، أو تخزين بيانات اعتماد، أو مزود دفع، أو مصدر حكومي، أو مزود AI، أو اتصال خارجي عام، قبل اجتياز جميع بوابات `P0` التالية.

| البوابة | الدليل المطلوب |
|---|---|
| الهوية | حسابات موثقة، وضبط جلسات، ومسار استعادة، واختبارات رفض تلقائية |
| عزل المستأجرين | حد منظمة على كل سجل عميل، واختبارات وصول عابر للمنظمات تفشل بإغلاق آمن |
| التفويض | مصفوفة أدوار وصلاحيات مفروضة من الخادم، وتدقيق لكل فعل مميز |
| الأسرار | لا سر في الشيفرة أو السجلات أو حالة المتصفح أو الأرشيف؛ تخزين وإجراء تدوير محددان |
| حماية البيانات | تصنيف للبيانات الحساسة، ونهج تشفير، وسياسة احتفاظ/حذف، وتجربة نسخ واستعادة |
| حماية API | تحقق من المدخلات، وحدود إساءة الاستخدام والحجم، ورؤوس آمنة، وسياسة أخطاء، وأثر تدقيق |
| التشغيل | فحوص صحة، ورؤية للحوادث، وسجلات تدقيق غير قابلة للتلاعب، وفحوص تكامل النسخ الاحتياطية |
| انضباط الإصدار | مراجعة اعتماديات، وفحوص أمنية آلية، وسجل إصدار معتمد، وإجراء رجوع |

اجتياز مراجعة بصرية وحدها لا يفتح الوصول الخارجي أبداً.

## 5. معنى «80% للعميل»

يجب أن يكمل الإصدار الأول القابل للتسويق عمل العميل الأساسي من نهايته إلى نهايته:

`اكتشاف ASIE -> إنشاء منظمة/مشروع -> إدخال أدلة ومدخلات -> تشغيل التحليل -> فهم KPIs والقرار -> التصرف تجاه المخاطر -> العودة إلى Snapshot/تقرير غير قابل للتغيير`

المقصود ليس 80% من كل ميزة يمكن تخيلها؛ بل 100% من حلقة القرار الحرجة.

### ما يشمله الإصدار الأول القابل للتسويق

- صفحة هبوط عربية أولا ومسار تحويل واضح.
- تسجيل دخول آمن وعضوية منظمة.
- إدخال موجه للمشروع.
- مركز قيادة للعميل تقوده مؤشرات الأداء.
- الأدلة والجاهزية والتشغيل والقرار والمخاطر والتنفيذ والتقرير وسجل Snapshot.
- مراجعة بشرية أساسية وتتبع للإجراءات.
- إدارة أساسية للمنظمة والمستخدمين والأدوار.
- عروض أساسية للمشغل: الصحة والتدقيق والدعم والاشتراك والاستخدام.
- حالة محلية للاشتراك والفواتير قبل أي بوابة دفع خارجية.

### ما يؤجل صراحة حتى يثبت طلب العملاء له

- معالج دفع خارجي.
- موصلات حية لمصادر حكومية أو بيانات مفتوحة.
- تكامل مزود AI.
- API عام للمطورين.
- `SSO/SAML` والتوفير المؤسسي المتقدم.
- محرك ضرائب متعدد العملات.
- محادثات آنية أو قنوات مراسلة خارجية.
- منشئ أدوار دقيق قابل للتخصيص.
- بنية موزعة أو تفكيك إلى microservices.

## 6. المسار A: الأمن والهوية وثقة البيانات

### PR-A0: معمارية الأمن ونموذج التهديد

**الهدف:** تعريف كيفية حماية الأشخاص والمنظمات والمشاريع والأدلة والقرارات وضوابط التشغيل قبل إضافة مزايا الوصول.

**المخرجات:**

- جرد الأصول وتصنيف البيانات: عامة، داخلية، سرية، ومقيدة.
- نموذج تهديد للمتصفح وAPI والمستودع وقاعدة البيانات والأرشيف وحدود مدير المنصة والتكاملات الخارجية.
- مخطط حدود ثقة يغطي الزائر المجهول والعضو الموثق ومدير المنظمة ومشغل الدعم ومدير المنصة.
- مصفوفة قبول أمنية وحالات إساءة الاستخدام.
- قرار سياسة كلمات المرور والجلسات والاستعادة.
- إجراءات النسخ والاستعادة والاستجابة للاختراق وتصعيد الحوادث.

**القبول:** لا تكتب شيفرة قبل تسمية مالك كل أصل وتهديده وتخفيفه وطريقة التحقق منه.

### PR-A1: أساس الهوية والمنظمات

**الهدف:** إضافة الحسابات والمنظمات بلا خلط لبيانات العملاء وبلا صلاحيات تتخذ في المتصفح.

**نطاق الحقيقة الأساسي:**

- حساب مستخدم.
- منظمة/مساحة عمل.
- عضوية.
- تعيين دور.
- جلسة.
- دعوة.
- رمز استعادة كلمة المرور.
- حدث تدقيق أمني.

**الأدوار الأولى:**

- مدير المنصة.
- دعم المنصة.
- مالك المنظمة.
- مدير المنظمة.
- محلل.
- مراجع.
- مشاهد.

**القبول:** كل مشروع وتشغيل وSnapshot وعنصر دليل ومراجعة وإجراء يصبح مقيداً بالمنظمة. يرفض الوصول أو الكتابة عبر المنظمات ويختبر ذلك.

### PR-A2: التفويض وتقوية API

**المهام:**

- طبقة سياسة/وسيط تفويض على الخادم.
- فحص ملكية المورد في كل مسار مشروع وSnapshot.
- تحقق من مخططات الطلبات وأغلفة أخطاء موحدة.
- معرفات للطلبات وبيانات تدقيق وحدود لحجم الحمولة ومعدلات الطلبات.
- رؤوس أمن المتصفح وسياسة CORS صارمة للأصل المحلي المقصود.
- منع تسرب stack trace أو الأسرار إلى الاستجابة أو سجل المتصفح.

**القبول:** اختبارات التفويض السلبية تغطي كل endpoint مميز؛ ولا يستطيع frontend منح نفسه صلاحية بإخفاء زر أو إظهاره.

### PR-A3: حماية البيانات والاستعادة

**المهام:**

- جرد الحقول الحساسة وقرار تشفير البيانات المخزنة.
- سياسة hashing لكلمات المرور وتخزين آمن لبيانات الاعتماد.
- صيغة النسخ الاحتياطي وتشفيرها والاحتفاظ بها والتحقق من استعادتها.
- سياسة الوصول إلى الأرشيف وإجراء التحقق من checksums.
- سير تصدير/حذف بيانات محكوم بالمنظمة والسياسة القانونية.

**القبول:** تنفيذ تجربة استعادة واحدة إلى بيئة محلية نظيفة ومقارنة بصمات سلامة Snapshot بعد الاستعادة.

### PR-A4: بوابة الجاهزية الأمنية

**القبول:** تمر كل اختبارات المسار A، وتوجد أدلة لكل بوابات `P0` في القسم 4، ويبقى الوصول الخارجي معطلاً إلى قرار إصدار صريح.

## 7. المسار B: طبقة إدارة المنصة

### PR-B0: نموذج نطاق طبقة التحكم

**الهدف:** بناء طبقة الحقيقة قبل رسم شاشات الإدارة.

**السجلات الأساسية:**

- المنظمات وحالاتها.
- المستخدمون والعضويات والأدوار والدعوات وأحداث الجلسات.
- خطة الاشتراك والاستحقاق والحصة ومقياس الاستخدام والفاتورة وحالة الدفع.
- حادث المنصة وحدث الصحة وحدث التدقيق وحالة الدعم والرسالة والإشعار.

**قاعدة:** حالة الفوترة أو الدفع هي سجل منصة، ولا يجوز لها إخفاء حقيقة جوهرية عن المخاطر أو القرار أو Snapshot من عميل مصرح له.

### PR-B1: عمليات المنصة

**المهام:**

- لوحة صحة مبنية على بيانات تشغيلية حقيقية للقراءة فقط.
- عرض لفشل الوظائف/عمليات التشغيل، وحالة النسخ الاحتياطية، والبحث في أحداث التدقيق، وزمن الحوادث.
- سير دعم مرتبط بمراجع المنظمة وSnapshot.
- نقل عرض معمارية runtime للقراءة فقط من التنقل العادي للعميل إلى لوحة المشغل.

### PR-B2: العمليات التجارية

**المهام:**

- نموذج محلي للخطط والاستحقاقات.
- تعريفات قياس الاستخدام.
- دفتر محلي للفواتير وحالة الدفع.
- دورة اشتراك: تجريبي، نشط، سماح، معلق، ملغى.
- تعديل يدوي للمشغل بسبب تدقيق غير قابل للمحو.

**مؤجل:** لا تحصيل دفع، ولا بوابة خارجية، ولا حساب ضرائب، ولا تخزين بيانات بطاقات في هذه المرحلة.

### PR-B3: أساس المراسلات والإشعارات

**المهام:**

- مركز إشعارات داخل المنتج.
- قوالب رسائل وحالة تسليم.
- إشعارات طلب المراجعة البشرية والاستجابة له.
- مراجع محادثة الدعم من دون كشف بيانات Snapshot السرية عبر المنظمات.

**مؤجل:** البريد الإلكتروني وSMS وWhatsApp والمزودات الخارجية إلى أن تسمح `PR-A4` بذلك.

### PR-B4: تجربة لوحة الإدارة

**تنقل المشغل:**

`نظرة عامة -> المنظمات -> المستخدمون والأدوار -> الاشتراكات -> الصحة -> التدقيق -> الدعم -> الإشعارات`

**القبول:** كل بطاقة وجدول يملك سجلاً خلفياً موثوقاً وقاعدة صلاحيات وحالة فارغة آمنة وأثر تدقيق لكل تغيير.

## 8. المسار C: تجربة المنتج والتصميم التجاري

### PR-C0: سرد المنتج ومعمارية المعلومات ونظام التصميم

**الهدف:** إنشاء لغة واحدة مترابطة للهبوط ومساحة العميل ولوحة الإدارة قبل تنفيذ الصفحات.

**المخرجات:**

- خريطة مصطلحات عربية أولا. تبقى المصطلحات الهندسية الإنجليزية في تفاصيل المشغل أو التشخيص فقط.
- شخصيات: زائر، مالك منظمة، محلل، مراجع، دعم منصة، مدير منصة.
- خريطة رحلة العميل ولحظات القرار.
- تسلسل المحتوى ولغة الحالات وحالات الفراغ والخطأ والتحميل.
- رموز التصميم: أدوار الألوان، الخطوط، المسافات، الارتفاعات، الحركة، لغة الرسوم، قواعد RTL، وقواعد الإتاحة.
- اتجاه بصري أصلي مستلهم من دفاتر تقنية وخرائط القرار وطبقات الأدلة.

**القبول:** يستطيع أي مستخدم تمييز الخطوة التالية وحالة القرار الحالية وثقة البيانات من دون فهم المعمارية.

### PR-C1: صفحة الهبوط ومسار التحويل

**الأقسام:**

- وعد ASIE ونتيجة القرار.
- معاينة KPI تفاعلية أو بصرية مبنية على بيانات مثال/عرض موسومة بوضوح.
- رحلة القرار: دليل -> تحليل -> قرار -> إجراء.
- إشارات الثقة والحوكمة: Snapshot غير قابل للتغيير، ومراجعة، ووضع محلي، وعدم اختلاق حكم من AI.
- حالات استخدام ودليل مناسب وإجراء تحويل.
- مسار واضح لتسجيل الدخول أو التجربة أو طلب الوصول وفق جاهزية المسار A.

**القواعد:** لا ادعاءات بعملاء أو وفورات أو دقة أو وصول رسمي للبيانات أو قدرة AI بلا دليل. تستخدم صور أصلية أو مرخصة أو من الملكية العامة فقط.

### PR-C2: إدخال العميل

**الرحلة:**

`إعداد المنظمة -> تعريف المشروع -> المدخلات -> الأدلة -> الجاهزية -> التشغيل`

**القبول:** يصل المستخدم لأول مرة إلى قرار أول صالح بلا مصطلحات تقنية ولا حقول إلزامية مخفية ولا تنقل مدمر.

### PR-C3: مركز قيادة العميل القائم على KPIs

**الهدف:** جعل أول شاشة بعد تسجيل الدخول منصة قرار احترافية، لا موقعاً قديماً ولا console تقنياً.

**قصة KPIs الأساسية:**

- حالة القرار ودرجة الثقة.
- NPV وIRR ومدة الاسترداد واحتياج التمويل واحتمال الجدوى عند توافرها.
- جاهزية البيانات/الأدلة وعدد العوائق الحرجة.
- التعرض للمخاطر والإجراء التالي المطلوب.
- آخر Snapshot غير قابل للتغيير وما تغير عن Snapshot السابق.

**السلوك:**

- الحركة تكشف التسلسل وحالة التغير؛ ولا تختلق تقدماً ولا تخفي نتيجة محظورة.
- الرسوم تستخدم بيانات Snapshot فعلية وتضع علامة واضحة للقيم غير المتاحة.
- إجراء أساسي واحد مشتق من حالة الجاهزية في الخادم.
- الانتقال إلى التفاصيل يفتح الأدلة والمخاطر والتنفيذ وخط النسب والتدقيق.

**القبول:** يستطيع المستخدم الإجابة في شاشة واحدة عن: «ما قراري الآن، ولماذا، وما الذي أفعله تالياً؟».

### PR-C4: مساحة قرار العميل

**الأقسام:**

- ثقة الأدلة.
- تبرير القرار.
- المخاطر وتخفيفها.
- خطة التنفيذ.
- المراجعة البشرية.
- إسقاطات التقرير وDecision Pack.
- خط Snapshot الزمني والمقارنة.

**قاعدة:** لا تعيد UI حساب منطق المال أو المخاطر أو القطاع أو الحكم. هي تعرض إسقاطات Snapshot وتوسم طبقات المراجعة بوضوح.

### PR-C5: ضمان الجودة المتجاوب والمتاح وعالي الثقة

**التحقق:**

- ترتيب القراءة العربي RTL.
- أساسيات لوحة المفاتيح وقارئ الشاشة.
- سطح مكتب `1440x900`، جهاز لوحي `1024x768`، وجوال `390x844`.
- حالات التحميل والفراغ والمنع وطلب الإدخال ورفض الصلاحية والخطأ.
- لا تداخل نصوص، ولا أيقونات مبهمة، ولا حالة مخفية، ولا إجراء مدمر بلا تنبيه.
- خط أساس للأداء المحلي وميزانية للأصول.

يجوز البحث والتصميم في المسارات بالتوازي، لكن لا تدعي أي حزمة الاكتمال قبل اجتياز تبعياتها.

## 10. تعريف اكتمال الإصدار الأول القابل للتسويق

- يفهم الزائر قيمة ASIE ويستطيع طلب أو بدء الوصول.
- العضو الموثق في منظمة يرى بيانات منظمته فقط.
- ينشئ العميل مشروعاً، ويدخل أدلة ومدخلات، ويشغل التحليل، ويحصل على مساحة قرار مفهومة تقودها KPIs.
- ترتبط كل نتيجة جوهرية بـ Snapshot أو إسقاط تقرير أو Decision Pack.
- يدير مسؤول المنظمة الأعضاء والأدوار.
- يدير مشغل المنصة حالة المنظمة والدعم والصحة وسجلات الاشتراك/الاستخدام المحلية.
- لا يفعل أي موصل خارجي من دون بوابة P0 وموافقة صريحة.
- يحتفظ بدليل قبول أمني وعزل مستأجرين ووظائف وواجهة وإتاحة.

## 11. ما ليس هدفاً في الإصدار الأول

- لا تضف كل ميزات المؤسسات قبل تجربة عميل.
- لا تبن واجهة فوترة شكلية بلا استحقاقات وسجلات تدقيق.
- لا تعرض معمارية AAS الخام للعملاء العاديين.
- لا تجعل لوحات KPIs معرض بطاقات زخرفياً.
- لا تستبدل حقيقة القرار المحكومة بنص AI.
- لا تفكك runtime المجمد إلى microservices لمجرد المظهر المؤسسي.

## 12. بوابات القرار

### البوابة 1: اعتماد المخطط

مطلوبة قبل تنفيذ `PR-01`:

- اعتماد سرد المنتج والعميل المستهدف.
- اعتماد نموذج التهديد الأمني.
- اعتماد مصفوفة الأدوار ونموذج المنظمة.
- اعتماد تعريفات KPIs للإصدار الأول.
- اعتماد معمارية معلومات صفحة الهبوط ومركز القيادة ولوحة الإدارة.

### البوابة 2: ألفا داخلية

مطلوبة قبل أي مستخدم خارجي:

- اجتياز `A1` حتى `A3`.
- وجود أساس `B0/B1` للعمليات.
- اجتياز إدخال العميل ومركز القيادة مراجعة سهولة استخدام داخلية.
- فشل كل اختبار وصول عابر للمنظمات.

### البوابة 3: تجربة محكومة

مطلوبة قبل عميل خارجي محدد:

- اجتياز كل بوابات الأمن `P0`.
- اجتياز تجربة الاستعادة.
- اختبار إجراءات الدعم والحوادث للمشغل.
- اعتماد محتوى الخصوصية والقانوني للعميل.
- تسجيل موافقة صريحة على الوصول الخارجي.
