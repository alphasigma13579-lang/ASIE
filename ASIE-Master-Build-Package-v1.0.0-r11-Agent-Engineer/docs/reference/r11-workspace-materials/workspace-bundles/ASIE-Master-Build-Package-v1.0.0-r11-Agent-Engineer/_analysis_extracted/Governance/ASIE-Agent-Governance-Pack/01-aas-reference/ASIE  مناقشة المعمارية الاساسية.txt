ASIE™ Operating Architecture
المعمارية التشغيلية لمنصة ASIE

Version: 1.0.0

1. الهدف

تحدد هذه الوثيقة كيفية عمل منصة ASIE أثناء التشغيل (Runtime).

لا تصف هذه الوثيقة:

لغة البرمجة.
قواعد البيانات.
Frameworks.
APIs.
الذكاء الاصطناعي.

بل تصف كيف تتحرك جميع أجزاء النظام أثناء التنفيذ.

2. فلسفة التشغيل

تعتمد ASIE على مبدأ بسيط:

النظام عبارة عن نواة ثابتة تدير مجموعة وحدات مستقلة عبر ناقل رسائل موحد.

أي أن النظام لا يتكون من خدمات مترابطة مباشرة، وإنما من وحدات مستقلة تتواصل عبر ناقل داخلي واحد.

سبب كل تسمية
التسمية	السبب
ASIE Constitution	أعلى مرجع حاكم للنظام.
ASIE Operating Architecture	يشرح فلسفة التشغيل وليس التنفيذ.
ASIE Platform Specifications	يجمع جميع مواصفات المنصة في مكان واحد.
ASIE Kernel Specification	مواصفات النواة.
ASIE Platform Protocol (APP) Specification	يحدد لغة التخاطب والعقود بين المكونات.
ASIE Runtime	البيئة التي تعمل فيها المكونات أثناء التشغيل.
ASIE Heart Controller	يدير القلوب وسياسات تشغيلها.
H1 / H2 / H3	هوية ثابتة للقلوب، أما الدور (Primary، Assist، Reserve) فيحدده الـ Controller ديناميكيًا.
ASIE Bus Controller	يدير تسجيل الوحدات، التوجيه، وسياسات الـ Bus.
ASIE System Bus	ناقل الرسائل المركزي.
Socket Contract Layer (SCL)	يفرض العقود على المنافذ قبل السماح بتركيب أي Module.
ASIE Modules	جميع الوحدات والإضافات التي تنفذ الوظائف.
تعديل واحد أنصح به بقوة

بدلاً من كتابة:

❤️ Primary
❤️ Assist
❤️ Reserve

اكتب:

❤️ H1
❤️ H2
❤️ H3

لأن:

H1/H2/H3 هي هوية ثابتة.
أما Primary / Assist / Reserve فهي حالة تشغيل يقررها ASIE Heart Controller، ويمكن أن تتغير أثناء عمل النظام.

بهذه الطريقة لا تربط هوية القلب بدوره، وتبقى المعمارية مرنة إذا احتجت مستقبلًا إلى تبديل الأدوار تلقائيًا حسب الحمل أو الأعطال، دون تغيير أي جزء من التصميم. وهذا يتوافق تمامًا مع فكرتك الأصلية عن ثلاثة قلوب تعمل حسب كثافة الاستخدام.

لماذا فصلنا Bus Controller؟

لأنه أصبح لدينا الآن مبدأ مهم جدًا:

الـBus ينقل الرسائل، أما الـController فيدير الناقل.

مثل عالم السيارات:

CAN Bus هو الأسلاك التي تنقل الرسائل.
أما وحدات التحكم فهي التي تقرر ماذا يُرسل ومتى وكيف.
مسؤوليات Bus Controller

هذه ستكون وثيقة مستقلة لاحقًا، لكن حتى نثبت المعمارية، هذه هي مسؤولياته:

1. Module Discovery

اكتشاف جميع الوحدات عند بدء التشغيل.

2. Registration

تسجيل الوحدات داخل النظام.

3. Contract Validation

التأكد من أن كل Module يحقق العقد المطلوب.

4. Version Compatibility

منع تركيب Module غير متوافق مع إصدار المعمارية.

5. Health Monitoring

متابعة حالة كل Module.

6. Hot Plug / Hot Unplug

إضافة أو إزالة Module أثناء تشغيل المنصة دون إعادة تشغيلها.

7. Message Routing Policies

تحديد سياسات التوجيه، بينما يبقى النقل الفعلي مسؤولية الـBus.

8. Isolation

عزل أي Module معطوب.

9. Recovery

إعادة تشغيل الوحدة أو استبدالها إذا لزم الأمر.

10. Dependency Resolution

التحقق من اعتماديات الوحدات قبل تحميلها.

أصبح لدينا الآن أربع طبقات تشغيلية واضحة
الطبقة	المسؤولية
Kernel	تشغيل النظام والتهيئة الأساسية
Heart Controller + Hearts	اتخاذ قرارات التشغيل وتوزيع الحمل
Bus Controller + System Bus	إدارة الوحدات ونقل الرسائل بينها
Modules	تنفيذ الوظائف الفعلية

لاحظ أن كل طبقة تعرف فقط الطبقة التي تليها، ولا تتجاوزها.

4. Kernel
المسؤولية

النواة هي مركز تشغيل النظام.

وظيفتها:

بدء تشغيل النظام.
تحميل الإعدادات.
تحميل العقود.
إنشاء الـBus.
إنشاء Heart Controller.
تسجيل الوحدات.

ولا تقوم بأي منطق أعمال.

5. Heart Controller

هو مدير التشغيل.

لا ينفذ الأعمال.

بل يقرر:

من يعمل؟
من ينتظر؟
من يتوقف؟
كيف توزع المهام؟
كيف يتم التعافي؟
مسؤولياته
مراقبة صحة القلوب.
مراقبة الحمل.
مراقبة الطوابير.
تفعيل القلوب.
إيقاف القلوب.
تبديل الأدوار.
إدارة الطاقة.
مراقبة استهلاك التوكنات.
مراقبة زمن الاستجابة.
6. القلوب الثلاثة

القلوب الثلاثة متطابقة وظيفياً.

ولا تختلف إلا في حالة التشغيل.

❤️ Primary Heart

الحالة الافتراضية.

يستقبل جميع الطلبات.

يوزع المهام.

يراقب النتائج.

❤️ Assist Heart

يبقى في وضع الاستعداد.

يعمل فقط عندما يطلب Heart Controller ذلك.

❤️ Reserve Heart

يبقى في وضع السكون.

يتحول إلى قلب نشط في:

الأعطال.
الصيانة.
الضغط المرتفع.
الكوارث.

8. ASIE System Bus

هو العمود الفقري للنظام.

كل رسالة تمر من خلاله.

ولا يسمح بأي اتصال مباشر بين الوحدات.

Discovery
Registration

9. Socket Layer

السوكيت ليس مكاناً لتركيب الوحدة فقط.

بل يمثل:

Contract

القيود.

12. مبدأ العزل

أي Module يجب أن يكون قابلاً للفصل دون التأثير على:

Kernel
Bus
Hearts
Modules الأخرى

ويعتبر أي اعتماد مباشر بين الوحدات مخالفة معمارية.

13. مبدأ الفشل

إذا فشلت وحدة:

Module

↓

Health Check Failed

↓

Bus Isolation

↓

Heart Notification

↓

Fallback

↓

Continue

لا يسمح بانهيار النظام بالكامل بسبب وحدة واحدة.

14. مبدأ التوسع

التوسع في ASIE لا يتم بإضافة كود إلى النواة، وإنما بإضافة وحدات جديدة أو استبدال وحدات قائمة، بشرط الالتزام بالعقود.

15. مبدأ الاستقلال

كل Module يجب أن يكون:

مستقلاً.
قابلاً للاختبار منفرداً.
قابلاً للاستبدال.
قابلاً للإيقاف دون إعادة تشغيل المنصة.
قواعد تشغيل إلزامية
لا اتصال مباشر بين الوحدات.
جميع الرسائل تمر عبر ASIE System Bus.
القلوب لا تعرف الوحدات، بل تعرف العقود فقط.
النواة لا تعرف مزودي الخدمات الخارجيين.
أي توسعة تتم بإضافة Module، وليس بتعديل Kernel.
أي عملية حسابية حتمية تنفذ داخل الوحدة المختصة، وليس داخل نموذج ذكاء اصطناعي.
ملاحظتي قبل الوثيقة الثالثة

أرى أن هناك مكونًا لم نعرّفه بعد، وهو Bus Controller. في الوثيقة الحالية أدرجت مسؤولياته داخل ASIE System Bus لتبسيط الصورة، لكن إذا أردنا بناء نظام على مستوى Enterprise فعلًا، فمن الأفضل أن نفصل بين:

ASIE System Bus: قناة الاتصال ونقل الرسائل.
Bus Controller: مدير الناقل، المسؤول عن تسجيل الوحدات، التحقق من العقود، تحميل الوحدات، مراقبة الصحة، وإدارة دورة حياتها.

هذا الفصل يجعل التصميم أنظف وأكثر قابلية للتوسع، وسنحسمه قبل كتابة الوثيقة الثالثة الخاصة بالـ Kernel و Heart Controller.

قرار الاعتماد الرسمي

Document ID: ASIE-ACP-000

Title:

ASIE Core Architecture v1.0 — Frozen

Status:

✅ Approved

Effective Date:

11 July 2026

ابتداءً من اليوم تصبح هذه قواعد المشروع
Rule 01

لا يجوز إضافة طبقة جديدة.

Rule 02

لا يجوز إضافة Controller جديد.

Rule 03

لا يجوز إضافة Bus جديدة.

Rule 04

لا يجوز إضافة Heart رابع.

Rule 05

لا يجوز تجاوز ASIE System Bus.

أي Module يتصل مباشرة بآخر يعتبر مخالفة معمارية.

Rule 06

لا يجوز لأي Module أن يعمل بدون Socket Contract.

Rule 07

لا يجوز لأي Module أن يتواصل خارج APP.

Rule 08

Kernel لا يعتمد على أي تقنية.

Rule 09

القلوب لا تنفذ Business Logic.

Rule 10

الـ Modules لا تعرف بعضها مباشرة.

ماذا يمكن تعديله مستقبلاً؟

يسمح فقط بتعديل:

إضافة Module جديدة.
إضافة Socket جديدة.
إضافة Plugin جديدة.
تعديل Contracts.
تحسين APP مع الحفاظ على التوافق (Backward Compatibility).
تحسين أداء القلوب.
تحسين Bus.

لكن دون تغيير الهيكل الأساسي.

ماذا يحتاج Architecture Change Proposal (ACP)؟

أي تغيير في:

Kernel
Heart Controller
Hearts
Bus Controller
ASIE System Bus
Socket Contract Layer

يتطلب ACP (Architecture Change Proposal)، مع تبرير فني ومراجعة رسمية، وليس مجرد قرار أثناء التطوير.
