# خطة تحويل بذرة ASIE إلى منتج محلي حقيقي

## الحالة الحالية
- توقفنا عند بذرة محلية تعمل: واجهة React/Vite + خلفية Python + عقود TypeScript + سياسة مصادر مغلقة افتراضيًا.
- التحقق الحالي نجح: فحص TypeScript بلا أخطاء، وملف الخلفية Python يُترجم بلا أخطاء.
- الموجود الآن يعرض مشروعًا تجريبيًا واحدًا ببيانات `DEMO_DATA`، وليس منصة كاملة بعد.
- لا يوجد مستودع Git صالح حاليًا رغم وجود مجلد `.git` فارغ/غير قابل للاستخدام.

## الملخص
- المرحلة القادمة ستكون: **مشروع كامل واحد end-to-end محليًا**.
- الأولوية: **الأساس الكامل للمنصة** لا مجرد واجهات.
- الواجهة التفصيلية ستستخدم **مسارًا مؤقتًا** من الحزمة إلى أن تزودني بمسار العميل الذي سترسمه.
- لا مفاتيح داخل الحزمة، ولا مصادر مرخصة/مدفوعة/حكومية غير مفتوحة. كل مصدر يبقى مغلقًا حتى يمر ببوابة مراجعة شروطه.

## تغييرات البناء الأساسية
- تحويل الخلفية من API تجريبي ثابت إلى نواة منصة محلية:
  - مشاريع متعددة.
  - لقطة تشغيل immutable snapshot.
  - سجل تدقيق لكل نتيجة.
  - سجل مصادر `Source Registry` بحالات: `candidate`, `blocked`, `enabled`, `reference_only`.
- تثبيت مسار AAS:
  - الواجهة لا تحسب ولا تستدعي AI ولا مصادر.
  - كل رقم يخرج عبر عقد typed envelope.
  - Finance Engine يملك الحسابات وMonte Carlo.
  - Decision Council ينسق الشخصيات الخمس والحكم السيادي بلا تصويت.
- إضافة تخزين محلي افتراضي:
  - SQLite للمرحلة المحلية.
  - طبقة Repository واضحة تسمح لاحقًا بالانتقال إلى PostgreSQL دون تغيير الواجهة.
- بناء مسار مستخدم مؤقت:
  - إنشاء مشروع.
  - إدخال بيانات أساسية.
  - اختيار عمق دراسة مبدئي.
  - تشغيل التقييم.
  - عرض الحكم، Monte Carlo، الشخصيات الخمس، العوائق، والأدلة.
  - توليد تقرير snapshot لا يعيد الحساب.
- إبقاء `pnpm` هو مدير الحزم، مع استمرار استخدام التشغيل المباشر للأدوات عند الحاجة إذا علّق wrapper الخاص بـ pnpm.

## الافتراضات المثبتة
- المرحلة القادمة محلية أولًا، وليست نشرًا إنتاجيًا عامًا.
- لا تسجيل دخول ولا صلاحيات مستخدمين في هذه المرحلة إلا إذا احتاجها مسار العميل لاحقًا.
- البيانات المفتوحة لا تعني “أي صفحة عامة”، بل dataset/API محدد ومسموح.
- الذكاء الاصطناعي لاحقًا يشرح فقط، ولا يملك أرقامًا أو أحكامًا أو تفسيرًا قانونيًا.
- مسار الواجهة الحالي مؤقت وقابل لإعادة الترتيب عندما تزودني برسم تجربة العميل.
 
# خطة مرحلة Core Engines v1 + Source Governance v1

## الملخص
- الهدف: نقل ASIE من ملف خلفية واحد بحسابات مبسطة إلى نواة محركات محلية منظمة، مع بقاء التشغيل محليًا وبدون جلب خارجي.
- النجاح يعني: مشروع واحد يمر عبر Finance Engine أوسع، Decision Council أوضح، Source Registry أكثر صرامة، وتقرير Snapshot غني يطابق الواجهة.
- سنحافظ على SQLite وPython القياسي وReact/Vite/pnpm، ولن نضيف مفاتيح أو مزودين أو مصادر خارجية في هذه المرحلة.

## تغييرات التنفيذ
- **تفكيك الخلفية داخليًا** داخل `backend/` إلى طبقات واضحة:
  - `repository`: التخزين وقراءة/حفظ المشاريع والتشغيلات واللقطات والمصادر.
  - `contracts`: بناء envelopes والأنواع المشتركة للخلفية.
  - `finance_engine`: الحسابات المالية وMonte Carlo و`NOT_READY`.
  - `decision_council`: الشخصيات الخمس والحكم السيادي الحتمي بلا تصويت.
  - `source_registry`: مراجعة المصادر وحالات `candidate / blocked / enabled / reference_only`.
  - `reports`: تقرير Snapshot لا يعيد الحساب.
- **توسيع Finance Engine** بإضافة:
  - NPV, IRR, payback, contribution margin, working-capital need, debt-service placeholder, sensitivity matrix.
  - سيناريوهات `baseline / conservative / optimistic`.
  - Monte Carlo أكثر صرامة: لا يعطي احتمالًا إذا غابت المدخلات أو التوزيعات أو seed أو convergence metadata.
- **تثبيت Decision Council v1**:
  - مخرجات مستقلة لكل شخصية: Project Manager, Business Advisor, Technical Auditor, Analyst Coach, Resistance Test.
  - كل شخصية ترى envelope مسموح فقط.
  - Sovereign Verdict يصدر من بوابات deterministic، وليس تصويتًا أو AI.
- **تقوية Source Governance**:
  - إضافة سجل مراجعة مصدر كامل: `terms_hash`, `license_snapshot_ref`, `classification`, `pdpl_check`, `nca_check`, `lawful_purpose`, `reviewer_decision`.
  - منع `enabled` إلا إذا اكتملت الحقول الإلزامية.
  - إضافة endpoint لعرض سجل المصادر وليس السياسة فقط.
- **تحسين الواجهة دون تغيير الحسابات فيها**:
  - عرض مؤشرات Finance الجديدة من الخلفية فقط.
  - عرض حالة كل شخصية وحالة Monte Carlo وسبب `NOT_READY`.
  - عرض ملخص مصدر/مراجعة المصدر داخل نفس المسار المؤقت.

## الواجهات والعقود
- تحديث/إضافة TypeScript contracts:
  - `FinanceResultSet`
  - `ScenarioResult`
  - `SensitivityMatrix`
  - `DecisionCouncilResult`
  - `PersonaInputScope`
  - `SourceReviewRecord`
  - `SnapshotReportSection`
- تحديث API المحلي:
  - `GET /api/sources`
  - `GET /api/projects/:id/runs`
  - `GET /api/runs/:runId/audit`
  - الإبقاء على endpoints الحالية دون كسر الواجهة.
- التقرير سيحتوي على: summary، KPIs، scenarios، sensitivity، personas، blockers، source governance، audit lineage.

## خطة الاختبار
- توسيع اختبارات Python لتغطي:
  - NPV/IRR/payback والحساسية.
  - `NOT_READY` عند نقص التوزيعات أو المدخلات.
  - عزل الشخصيات ومنع التصويت.
  - رفض تفعيل مصدر ناقص المراجعة.
  - تطابق التقرير والواجهة مع نفس snapshot.
- اختبارات حراسة r10/r11:
  - لا حساب مالي داخل React.
  - لا مصدر خارجي مفعّل افتراضيًا.
  - لا AI أو provider يملك رقمًا أو حكمًا.
  - لا Dashboard/Feasibility/Procurement module جديد كمالك حقيقة.
- فحوصات محلية نهائية:
  - `python -m unittest discover -s tests`
  - `python -m py_compile`
  - TypeScript check
  - Vite build
  - API smoke test لمسار مشروع كامل.

## الافتراضات
- سنبقى محليين على المنفذ `8788` لتجنب التصادم الحالي مع `8787`.
- لن نضيف FastAPI أو PostgreSQL الآن؛ التنظيم الداخلي يكفي لهذه المرحلة.
- لا نضيف مصادر بيانات فعلية أو جلب خارجي حتى تزودني بمصدر محدد ونراجع شروطه.
- الواجهة المؤقتة تبقى قابلة لإعادة الترتيب عندما ترسم مسار العميل النهائي.


# خطة المرحلة التالية: Product Workflow v1 + Evidence & Report Pack

## الملخص
- الهدف: تحويل المحركات الحالية إلى تجربة منتج عملية للمستخدم: مسار إدخال منظم، دفتر افتراضات، حوكمة مصادر مرئية، وتقرير Snapshot قابل للاعتماد محليًا.
- السبب: لدينا الآن Core Engines وSource Governance، لكن المستخدم ما زال يدخل أرقامًا مباشرة في شاشة واحدة، والتقرير موجود كبيانات JSON وليس كحزمة قرار مفهومة للعميل.
- مخرج المرحلة: مشروع واحد يمر عبر Wizard واضح، ينتج Assumption Book وEvidence Register وSnapshot Report غني داخل الواجهة، مع بقاء كل الحسابات في الخلفية.

## تغييرات التنفيذ
- **Project Wizard v1**
  - تقسيم المسار المؤقت إلى خطوات فعلية: تعريف المشروع، نموذج الإيرادات، التكاليف، التمويل، الافتراضات، المصادر، المراجعة، التشغيل.
  - حفظ المشروع كمسودة قبل التشغيل، وإظهار `NOT_READY` لكل خطوة ناقصة بدل السماح بتشغيل مبهم.
  - إبقاء الواجهة كمنسق عرض فقط؛ لا حسابات مالية في React.

- **Assumption Book v1**
  - إضافة سجل افتراضات لكل مشروع: `assumption_id`, `label`, `value`, `unit`, `owner`, `source_type`, `confidence`, `review_status`.
  - ربط كل KPI بالافتراضات المستخدمة بدل `assumption:local-user-inputs-v1` العامة.
  - إضافة واجهة تعرض الافتراضات وتبين ما هو مدخل مستخدم، افتراضي محلي، أو يحتاج مراجعة.

- **Evidence & Source Workbench v1**
  - توسيع Source Registry ليظهر داخل الواجهة كسجل مراجعة لا كمجرد عدادات.
  - إضافة شاشة مصدر: الحالة، سبب الحجب/الترشيح، حقول المراجعة المطلوبة، وقرار المراجع.
  - لا جلب خارجي، ولا تفعيل تلقائي. المصدر المفعّل يعني “مسموح لاحقًا” وليس “تم الجلب الآن”.

- **Snapshot Report v1**
  - تحويل التقرير من ملخص صغير إلى تقرير قرار داخل الواجهة: ملخص تنفيذي، KPIs، السيناريوهات، الحساسية، الشخصيات، العوائق، الافتراضات، المصادر، والتدقيق.
  - إضافة endpoint لتصدير تقرير HTML محلي من نفس snapshot، بدون إعادة حساب.
  - إظهار `snapshot_id`, `run_id`, `algorithm_versions`, و`audit_refs` بوضوح.

## خطة الاختبار

  - تقرير HTML يطابق نفس snapshot ولا يعيد الحساب.

  - لا توجد معادلات مالية أو Monte Carlo داخل React.
- فحوصات نهائية:
  - Python unit tests.
  - Python compile.
  - TypeScript check.
  - Vite build.

# خطة المرحلة التالية: Operational Finance Depth v1

## Summary
- الهدف: تعميق الدراسة ماليًا وتشغيليًا بدل إضافة واجهات فقط.
- نوسّع Finance Engine من نموذج أرقام عامة إلى نموذج تشغيل أولي: طاقة تشغيل، استخدام، OPEX مفصل، CAPEX مفصل، خدمة دين، ونقاط ضغط.
- تبقى القواعد الحالية ثابتة: لا حسابات في React، لا جلب خارجي، لا مفاتيح، كل نتيجة داخل snapshot immutable، والتقرير يقرأ نفس snapshot.

## Key Changes
- توسيع مدخلات المشروع:
  - إضافة حقول تشغيلية: `use_operating_capacity`, `capacity_units_per_day`, `operating_days_per_month`, `utilization_rate`.
  - إضافة OPEX مفصل: `payroll_monthly`, `rent_monthly`, `utilities_monthly`, `marketing_monthly`, `maintenance_monthly`.
  - إضافة CAPEX مفصل: `capex_equipment`, `capex_fitout`, `capex_licenses_local`, `depreciation_years`.
  - إضافة تمويل أعمق: `equity_contribution`, `loan_grace_months`, مع إبقاء `debt_amount`, `annual_interest_rate`, `loan_years`.
- تحديث Finance Engine:
  - إذا `use_operating_capacity=true` يشتق `monthly_units = capacity_units_per_day * operating_days_per_month * utilization_rate`.
  - إذا لم تُفعّل، يبقى `monthly_units` اليدوي هو المصدر كما هو الآن.
  - حساب OPEX total من البنود الجديدة مع دعم الحقول القديمة للتوافق.
  - حساب CAPEX total، depreciation monthly، EBITDA، EBIT، net operating cashflow، DSCR، funding need after equity.
  - إضافة sensitivity جديدة: utilization × price، وOPEX × demand.
- تحديث Decision Council:
  - Project Manager يقرأ readiness/capacity/funding gap.
  - Business Advisor يقرأ contribution margin وutilization وDSCR.
  - Technical Auditor يقرأ اكتمال نموذج التشغيل والحساسية.
  - Resistance Test يضغط utilization وOPEX وdebt service.
  - الحكم السيادي يبقى deterministic ولا يوجد تصويت.
- تحديث الواجهة والتقرير:
  - إضافة قسم “نموذج التشغيل والتمويل” داخل المسار الحالي.
  - عرض مؤشرات التشغيل والتمويل من الخلفية فقط.
  - تحديث Snapshot Report ليشمل operational KPIs، CAPEX/OPEX breakdown، DSCR، وutilization sensitivity.

## Public APIs / Types
- تحديث `ProjectInputs` و`FinanceResultSet` و`ScenarioResult` في TypeScript لإضافة الحقول والمؤشرات الجديدة.
- لا نكسر endpoints الحالية.

  - `opex_breakdown`
  - `debt_service_profile`
  - `operational_sensitivity`

  - صحة OPEX/CAPEX totals وdepreciation وEBITDA/EBIT.

  - Snapshot قديم لا يتغير بعد تعديل مدخلات التشغيل.
  - Decision Council لا يستخدم تصويتًا ولا AI.

  - TypeScript check.
  - Vite build.
- Smoke test:
  - draft → تشغيل capacity-based run → report-view → report HTML.

## Assumptions
- الإدخال سيكون محليًا يدويًا فقط في هذه المرحلة.
- لا يتم الاتصال بأي مصدر خارجي أو حكومي.
- ربط الدليل لا يعني أن المصدر رسمي أو مفعّل إلا إذا اجتاز بوابات Source Registry وDataset Quality Gate.
- Finance Engine وDecision Council يقرآن evidence coverage كحالة داعمة فقط؛ لا يصبح الدليل مالكًا للحسابات أو الحكم.
