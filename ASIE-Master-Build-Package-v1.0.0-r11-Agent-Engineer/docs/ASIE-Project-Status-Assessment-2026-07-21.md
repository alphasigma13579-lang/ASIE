## ما اكتمل فعلاً

### 1. النواة المعمارية

- المسار المحكوم يمر عبر Kernel/Hearts/Bus/Socket/Project Run ثم Snapshot Assembly.
- كل تشغيل ينتج Snapshot ثابتاً؛ التقرير وDecision Pack يقرآن منه دون إعادة حساب في React.
- توجد اختبارات للتلاعب، الهوية بين Run وSnapshot، الكتابة الذرية، idempotency، وثبات اللقطات القديمة.
- AAS Runtime مجمد وفق Manifest؛ أي تغيير لاحق يحتاج ACR.

### 2. التحليل والقرار

- Finance، Sector Intelligence، Evidence، Decision Council، Risk وExecution متصلة بعقود مغلقة.
- توجد بوابات NOT_READY ومنع لاختلاق قيمة عند نقص المدخلات.
- Monte Carlo والمخاطر وخطة التنفيذ محفوظة ضمن الإسقاطات، لا تحسب في الواجهة.

### 3. الأدلة والبيانات المحلية

- CSV/XLSX، فحص الجودة، مراجعة المصدر، التحويلات، Evidence Ledger وLineage موجودة.
- المصدر غير المراجع أو غير المرخص لا يصبح دليلاً معتمداً تلقائياً.
- التعديلات بعد Snapshot لا تغير التقرير القديم.

### 4. الهوية والأمن المحلي

- كلمات المرور مشتقة بـPBKDF2، ورمز الجلسة الخام لا يخزن.
- العضوية والمنظمة والدور تتحقق في الخادم.
- استعادة الحساب المحلية أصبحت موجودة: رمز لمرة واحدة، انتهاء، إبطال الجلسات، وإعادة ضبط إداري.
- Origin محلي، حد حمولة، rate limit، رؤوس أمنية، request id وسجل audit موجودة.

### 5. التقارير

- تقرير تمويلي من 16 قسماً مربوط بنفس Snapshot.
- HTML تمويلي موجود عبر `/api/snapshots/{snapshot_id}/funder-report.html`.
- مولدات `DOCX` و`PDF` و`PPTX` موجودة في `backend/report_exports.py`.
- PPTX تحقق بنيوياً كحزمة OpenXML، وPDF تحقق كبنية PDF، لكن ذلك لا يعني أن العميل يستطيع تنزيلهما من الواجهة الآن.

## ما هو جزئي أو غير مكتمل

### 1. رحلة تعريف المشروع — الفجوة الأعلى أولوية

المسار المقصود صحيح: الموقع → القطاع → التصنيف → الاسم → الفجوة والميزة → الجمهور → رأس المال → التفاصيل. التنفيذ الحالي ما زال يعاني من الآتي:

- السؤال الفعلي منفصل في DOM عن رأس المعالج وشريط المراحل، فينشأ تباعد بصري وإحساس بصفحتين داخل خطوة واحدة.
- الموقع يحتوي حقلاً عاماً وأربعة حقول إضافية؛ حقول الدولة والمنطقة والمدينة والشارع تكتب فوق `location_scope` نفسها بدلاً من حفظ عنوان منظم.
- ضغط GPS يستدعي `navigator.geolocation`، لكن الخادم يرسل `Permissions-Policy: geolocation=()`؛ لذلك السلوك متناقض وغير صالح حالياً.
- اختيار القطاع والتصنيف ينتقل تلقائياً، لكن خيار «قطاع آخر» و«تصنيف آخر» يعرض رسالة فقط ولا يفتح إدخالاً ولا يحفظ خياراً.
- المساعدة في الأسماء والفجوة معطلة؛ لا توجد خدمة اقتراح محلية أو مزود AI معتمد.
- القيم الافتراضية المالية والمشروع الافتراضي قد تبدو كبيانات المستخدم إذا لم تميز بوضوح.
- ما زالت صفحات الأدلة والجاهزية والقرار تعرض مصطلحات مثل Dataset وLedger وLineage وSnapshot وMonte Carlo وNOT_READY للعميل مباشرة.

### 2. التصدير

- لا توجد endpoints ظاهرة لـ `.docx` أو `.pdf` أو `.pptx` في `asie_local_api.py`.
- لا توجد أزرار تنزيل لهذه الصيغ في React؛ الموجود هو فتح تقرير HTML.
- اختبار DOCX متخطى اليوم بسبب عدم توفر مكتبة التصدير/الفحص.
- PDF يعتمد على renderer خلفي أو fallback تطويري؛ لا توجد خدمة إنتاجية مثبتة ومراقبة.
- PPTX مولد أساسي، ولم يمر بعد بمراجعة بصرية شاملة على PowerPoint/LibreOffice/Keynote.

### 3. العزل والأمن الخارجي

- توجد اختبارات HTTP صريحة لمنع قراءة مشروع عبر منظمة أخرى ومنع انتحال المنظمة عند إنشاء مشروع.
- لا يوجد دليل اختبار route-by-route يغطي كل project/run/snapshot/dataset/report/admin endpoint بمستخدم من منظمة أخرى.
- النسخ والاستعادة متحققة بنيوياً، لكن لا توجد جدولة فعلية أو سياسة احتفاظ منفذة أو مراقبة فشل النسخ.
- التشفير مؤجل، ولذلك يبقى تصدير النسخ خارج الجهاز ممنوعاً.
- لا توجد مراجعة SBOM/dependency audit موثقة حالياً.

### 4. البيانات الحقيقية والخرائط

- Live Cockpit وخرائط المنافسين بيانات `DEMO / LOCAL ONLY`.
- لا توجد Google Maps/Mapbox أو geocoding أو منافسون حقيقيون أو سياسة cache/تكلفة/خصوصية للمزود.
- قائمة المصادر تحتوي عناوين مرجعية، لكنها لا تنفذ fetch خارجي، وهذا صحيح أمنياً لكنه يعني أن المنتج لا يملك سوقاً حياً بعد.

### 5. التشغيل التجاري

- entitlement، usage، invoice محلي، وإشعارات داخل المنتج موجودة كسجلات خلفية.
- لا توجد بوابة دفع، ضريبة، تجديد، فشل دفع، بريد، SMS، WhatsApp، أو دعم خارجي.
- لوحة المشغل تعرض البيانات، لكن workflow الإدارة الكامل والتفويض التشغيلي في بيئة متعددة المؤسسات لم يثبت بعد.

## حالة التشغيل الآن

- API على المنفذ `8794` كان يعمل وقت المراجعة وأعاد `/api/health` بنجاح.
- الواجهة على `5194` و`5195` لم تكن تعمل وقت الفحص.
- عملية API الجارية تبدو أقدم من الكود الحالي: مسار `/api/operations/release-info` موجود في المصدر لكنه أعاد `404` من الخدمة العاملة. يلزم إيقاف العملية القديمة وتشغيل backend الحالي قبل الاعتماد على أي فحص واجهة.
- Git CLI غير متاح في PATH، لذلك لم يمكن استخراج diff موثوق أو حالة worktree. التقييم مبني على الملفات الحالية والاختبارات، لا على مقارنة commit.

## خطة الإغلاق الموصى بها

### المرحلة 1 — UX-Flow v1: رحلة عميل واحدة

الهدف: أول مستخدم يفهم البداية والنهاية دون شرح خارجي.

معايير القبول:

1. سؤال واحد ومحتوى واحد في كل شاشة، ولا توجد لوحة إضافية أسفل الصفحة.
2. عنوان موقع منظم: دولة، منطقة، مدينة، حي/شارع، وإحداثيات اختيارية.
3. توافق سياسة GPS مع موافقة المستخدم، أو إزالة الزر حتى اعتمادها.
4. قطاعات وتصنيفات كبطاقات، مع إدخال مخصص حقيقي ومراجعة قبل إضافته للقاموس العام.
5. انتقال تلقائي فقط للاختيارات القطعية، و«التالي» للنصوص والمراجعة.
6. لغة عربية للعميل؛ التفاصيل التقنية تنتقل إلى «تفاصيل المشروع» أو «التدقيق».
7. اختبار تفاعلي على 1440 و1024 و390، واختبار لوحة مفاتيح وحالات الخطأ والتحميل.

### المرحلة 2 — Export Delivery v1

الهدف: يستطيع العميل تنزيل تقريره فعلياً.

معايير القبول:

1. endpoints مصرح بها لـHTML/PDF/DOCX/PPTX/XLSX من نفس projection.
2. أزرار تنزيل واضحة مع حالة توليد وفشل.
3. تثبيت renderer PDF خلفي واحد وخطوط عربية مع health check.
4. تثبيت بيئة DOCX وإزالة skip.
5. مراجعة بصرية لـPPTX على PowerPoint وLibreOffice، وربط الملف بـSnapshot/hash.

### المرحلة 3 — Tenant Security Matrix

الهدف: إثبات عدم تسرب البيانات قبل أي عميل خارجي.

معايير القبول:

1. جدول لكل endpoint والـpermission المطلوب.
2. اختبارات رفض cross-tenant لكل project/run/snapshot/dataset/report/review/action/admin route.
3. اختبارات session expiry/revocation/recovery والتلاعب برؤوس المنظمة.
4. دليل audit لكل محاولة مرفوضة من دون تسريب محتوى.

### المرحلة 4 — Operational Release

الهدف: خدمة قابلة للاستعادة والمراقبة.

معايير القبول:

1. تشغيل موحد للواجهة والخلفية من إصدار واحد، مع منع العمليات القديمة.
2. جدولة backup، فحص دوري، تنبيه فشل، وتمرين restore موثق.
3. SBOM وفحص الاعتمادات وخطة rollback مجربة.
4. health/readiness، سجلات أخطاء، مراقبة renderer والتخزين.
5. قرار منفصل للتشفير قبل خروج أي نسخة من الجهاز.

### المرحلة 5 — Pilot حقيقي محدود

الهدف: إثبات قيمة المنتج قبل الخرائط والدفع.

معايير القبول:

2. قياس إكمال رحلة الإدخال، الزمن لأول تقرير، نقاط الانسحاب، وفهم القرار.
3. مراجعة محاسب/تمويل مستقلة لعينة التقارير.
4. إصلاح الملاحظات الحرجة قبل فتح الخرائط أو الدفع.
