# ASIE — إعادة تخطيط المنتج وتجربة العميل: المستشار الاستراتيجي التفاعلي

## قرار المنتج

تتحول ASIE من واجهة تقرير وجدوى إلى **مقر عمل يومي للمشروع**: مستشار استراتيجي يربط القرار الحتمي بالسوق والسياق والتنفيذ.

لا يشتري العميل KPI أو PDF؛ يريد إجابة عملية: هل أمضي ولماذا؟ أين أقف أمام الواقع؟ ما المخاطر والفرص التي لا تظهر في المعادلة؟ وما الخطوة التالية؟

لا تعد ASIE باليقين؛ تقدم **ثقة قابلة للتتبع**: حقيقة مختومة، سياق، دليل، افتراض، فجوة، وإجراء.

## ما يبقى ثابتاً معماريّاً

```text
مدخلات + أدلة
  → محركات المال/القطاع/القرار/المخاطر/التنفيذ الحتمية
  → Snapshot Assembly غير قابل للتغيير
  → إسقاطات العميل: ذكاء سياقي + اختبار واقع + توصيات + تقدم تنفيذ
```

| الطبقة | المسؤولية | ممنوع عليها |
|---|---|---|
| AAS Runtime المجمد | الحساب، الحكم، المخاطر، التنفيذ، Snapshot | تعديل من تجربة العميل أو widget |
| Contextual Intelligence | Benchmark، Macro، منافسون، موقع، إرشاد | تغيير الحكم أو حساب المال محلياً |
| Advisory/Execution Overlay | توصيات، مالك، موعد، تقدم، ملاحظات مراجعة | تعديل Snapshot أو Sovereign Verdict |
| واجهة العميل | فهم، استكشاف، drill-down، تنفيذ الخطوة | امتلاك أرقام محكومة أو مصدر غير موثق |

الشخصيات الخمس وMonte Carlo إسقاطات قرار مرئية في Dashboard: تفسر زوايا القرار ومحاكاة المخاطر، ولا تصوّت ولا تحل محل الحكم السيادي.

## قفل نطاق تنفيذ UX-R1

هذه الدفعة تعيد ترتيب **غرف الواجهة فقط**: الصفحة التي يدخلها العميل، ما يراه في كل مقصد، التنقل، الحركة، والأثاث البصري.

- لا تعديل في AAS Runtime أو المحركات أو الحسابات أو Snapshot أو عقود API.
- لا نموذج بيانات خلفي جديد في هذه الدفعة؛ أي بيانات تطويرية تبقى fixture عرض محلياً وموسومة.
- لا تحمل React حكماً أو جاهزية أو Monte Carlo أو حساباً مالياً؛ تعرض الإسقاطات الموجودة فقط.
- التغيير المسموح: صفحات React، مكونات العرض، CSS، وسجل التنقل في المتصفح.

## سياسة بيانات التطوير المحلية

بيئة التطوير محلية ومغلقة. يسمح فيها باستخدام بيانات محاكاة غنية لبناء واختبار المنتج كاملاً، بما في ذلك المنافسون والمواقع والمشاريع المرجعية والمؤشرات الكبرى ومسارات الدعم والتراخيص.

كل سجل محاكى يحمل إلزامياً:

```text
data_mode: demo_simulated_external
display_badge: DEMO / LOCAL ONLY
assumed_source: المصدر المفترض أو فئته
observed_at: تاريخ seed
confidence: تجريبي
production_admission: blocked
```

- لا يوجد اتصال فعلي أو Google Maps أو GPS أو مصدر حكومي أو AI provider في هذه المرحلة.
- لا تدخل بيانات المحاكاة في Snapshot أو Report أو قرار إنتاجي.
- قبل الإطلاق الخارجي تزال أو تستبدل جميعها بسجلات مصدر مرخصة ومراجعة.
- يمنع عرض مشروع منظمة أخرى في «مشاريع مشابهة» إلا عبر كتالوج موافق عليه ومزال الهوية.

## نموذج تجربة العميل الجديد: صفحات منطقية لا صفحة متكدسة

**قرار إلزامي:** لا تبنى تجربة العميل كلوحة واحدة تخفي/تظهر عشرات الأقسام. كل مقصد عمل صفحة مستقلة لها عنوانها وسياقها وإجراءاتها. لا تبقى KPIs والخريطة والمعالج والتقارير ظاهرة معاً.

```text
01 لوحة القيادة الرئيسية
   ├─ 02 مشروع جديد / Smart Input Hub
   │   ├─ 03 اختبار الواقع والسوق
   │   ├─ 04 قرار المشروع
   │   └─ 05 خارطة التنفيذ
   └─ 06 تقاريري وسجل Snapshots
```

| الصفحة | سؤال العميل | المحتوى المسموح فقط | الإجراء الرئيسي |
|---|---|---|---|
| لوحة القيادة | ما وضع مشاريعي وما يحتاج انتباهاً؟ | KPI مختصر، تنبيه، آخر قرار، المشاريع | افتح مشروعاً أو ابدأ مشروعاً |
| مشروع جديد/Smart Input | ما الذي تريد بناؤه وأين؟ | نشاط، قطاع، نطاق، هدف، أدلة مطلوبة | حفظ والانتقال لاختبار الواقع |
| اختبار الواقع والسوق | هل الفرضية تصمد أمام السوق والسياق؟ | Benchmark، Macro، منافسون، موقع، فجوات دليل | أكمل دليل أو تابع للقرار |
| قرار المشروع | هل أمضي ولماذا؟ | الحكم، Monte Carlo، الشخصيات الخمس، المخاطر | افتح خارطة التنفيذ |
| خارطة التنفيذ | ماذا أفعل من اليوم؟ | Actions، دعم، إجراءات، تقدم، عوائق | تحديث التقدم أو فتح دليل الإجراء |
| تقاريري | ماذا وثقنا وما الذي تغير؟ | Report، Decision Pack، Snapshots، مقارنة | فتح/تصدير إسقاط محفوظ |

### قواعد الانتقال

- الانتقال إلى خطوة تالية: الصفحة الجديدة تدخل من **اليسار** في RTL، والسابقة تخرج باتجاه اليمين.
- العودة: الصفحة السابقة تدخل من **اليمين**، والحالية تخرج إلى اليسار.
- يستخدم زر المتصفح Back/Forward وسجل المسار نفسه، ولا يفقد `project_id` أو حالة المسودة.
- يحمل كل مسار `project_id` عند ارتباطه بمشروع؛ لا تعتمد الصفحة على state عابر فقط.
- تحترم الحركة `prefers-reduced-motion` وتبقى قابلة للوصول بلوحة المفاتيح.
- لا تبقى صفحة سابقة متكدسة أسفل الصفحة التالية؛ mount/unmount أو transition shell فقط.

## التنقل العلوي

1. **قراري اليوم** — الحكم، السبب، Monte Carlo، الشخصيات الخمس، الإجراء الأهم.
2. **اختبار الواقع** — Benchmark cases، Macro context، المنافسون، المواقع المرشحة.
3. **خارطة التنفيذ** — خطة الخطوات، الدعم والإجراءات، المالك، الموعد، التقدم والعوائق.
4. **تقاريري** — Report، Decision Pack، Snapshots والمقارنة.

المعالج التقني والأدلة التفصيلية لا يختفيان، لكنهما خطوات داخل صفحات المشروع، وليسا محور Dashboard.

### قواعد UI/UX

- كل Widget مستقل في العرض والإخفاء وإعادة الترتيب، لكن ليس في الحقيقة الخلفية.
- لكل Widget مصدر، وقت تحديث، badge بيانات، وحالة empty/blocked/error مفهومة.
- كل KPI قابل للضغط للـdrill-down إلى الدليل أو الافتراض أو الخطر.
- لا drag-and-drop حصري بالفأرة؛ يجب توفير Move earlier/later بلوحة المفاتيح.
- لا خريطة أو KPI زخرفي بلا badge يوضح مصدره وحالة بياناته.

## خطة العمل المرحلية

### UX-R0 — تثبيت العقود وخطة الهجرة

- اعتماد الوثيقة ومعجم المصطلحات العميلية.
- حصر كل عنصر ظاهر: إبقاؤه، نقله إلى تفاصيل، أو إزالته من العميل.
- إنشاء fixtures محلية `DEMO / LOCAL ONLY` لكل نموذج جديد.
- القبول: لا تغير في Runtime/Snapshot، وكل fixture معزول عن بيانات الإنتاج.

### UX-R1 — إعادة تخطيط Shell وتجربة القرار

- بناء route/page shell مستقل وسجل browser history وحركة RTL أمام/خلف.
- جعل لوحة القيادة صفحة مستقلة، ثم «قراري اليوم» داخل صفحة المشروع.
- نقل Monte Carlo والشخصيات الخمس إلى وحدة ذكاء القرار.
- فصل المعالج والأدلة واختبار الواقع والتنفيذ والتقارير في صفحات منفصلة.
- القبول: لا يعرض أي مسار محتوى صفحة أخرى، والعودة تحفظ سياق المشروع.

**حالة التنفيذ الحالية:** بُني Shell للصفحات المقصودة والتنقل الأمامي/الخلفي RTL وسجل المتصفح، وفصلت صفحة «اختبار الواقع» و«خارطة التنفيذ» عن لوحة القيادة. يبقى ترحيل أدوات الإدخال والأدلة التفصيلية من الـbuilder القديم إلى غرفها الجديدة عملاً لاحقاً داخل UX-R1، من دون تغيير خلفي.

### UX-R2 — Live Cockpit والمحاكاة المحلية

- Smart Input Hub وWidgets مستقلة وإخفاء/إظهار وترتيب قابل للوصول.
- خريطة SVG/Canvas محلية ومنافسون ومواقع محاكاة.
- Reality Benchmark وMacro Context محاكيان وموسومان.
- القبول: كل تفاعل يعمل محلياً ولا يوجد طلب شبكة خارجي.

**حالة التنفيذ 2026-07-20:** نُفذت صفحة «اختبار الواقع» كغرفة سوق مستقلة: خريطة محلية تفاعلية، إشارات للتحقق، نماذج مقارنة محلية، سياق كلي محاكى، وخطوات لتحويل الإشارة إلى دليل. كل وحدة ظاهرة تحمل `DEMO / LOCAL ONLY` ومصدرها المفترض وزمن seed المحلي ودرجة الثقة التجريبية. لا يوجد تعديل خلفي أو طلب شبكة أو كتابة إلى Snapshot.

### UX-R3 — Execution Compass والإرشاد

- خطة تنفيذ مرتبطة بـSnapshot كـoverlay مستقل.
- Guidance Catalog للدعم والتراخيص والتأشيرات مع disclaimer ومراجعة.
- كتالوج مشاريع مشابهة بموافقة/محاكاة فقط.
- القبول: تحديث التقدم لا يغير Snapshot، ولا كشف عابر للمنظمات.

### UX-R4 — دمج المصادر الحقيقية لاحقاً

- لا يبدأ قبل Gate P0 وموافقة صريحة.
- adapters منفصلة لكل مصدر/خريطة/AI provider، مع policy/review/audit.
- استبدال fixtures بسجلات موثقة مع الاحتفاظ بوسم المصدر والتاريخ والحدود.

## معايير القبول الشاملة

1. لا حساب مالي أو قرار أو Monte Carlo أو تقييم موقع في React.
2. الحكم السيادي وSnapshot يبقيان الحقيقة غير القابلة للتغيير.
3. الشخصيات الخمس وMonte Carlo ظاهران ومفسران لكن لا يملكان vote.
4. كل data widget يوضح `source`, `as_of`, `data_mode`, و`confidence`.
5. بيانات المحاكاة لا تستطيع المرور إلى إنتاج أو إخفاء badge الخاص بها.
6. لا وصول خارجي في R0–R3.
7. واجهة العميل متجاوبة ومفهومة على 1440x900 و1024x768 و390x844.
8. كل حالة فارغة تقول ما البيانات الناقصة وما الخطوة التالية.
9. كل صفحة عميل لها مقصد واحد؛ لا توجد صفحة Dashboard طويلة تجمع كل المنتج.
10. الانتقال الأمامي/الخلفي يحترم RTL وbrowser history وreduced motion.

## قرار التنفيذ التالي

يبدأ التنفيذ من **UX-R0 ثم UX-R1**. لا يبدأ بناء أي adapter خارجي أو تشغيل Google Maps/GPS/AI provider ضمن هذه الخطة.
