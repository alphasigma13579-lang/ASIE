# ASIE — قرار حزمة التقرير الجاهز للتمويل وخطة العمل

**المعرّف:** `ASIE-FRRP-01-2026-07-20`  
**الحالة:** `DRAFT — قرار مطلوب قبل التنفيذ`  
**القرار المقترح:** لا نبني «مولد PDF» منفصلاً؛ نبني إسقاط تقرير تمويلي من Snapshot محفوظ، ثم نضيف قوالب الجهات والقطاعات كطبقة فوقه.  
**نطاق هذه الوثيقة:** مراجعة الواقع الحالي، مرجع التقرير، الفجوات، وخطة العمل. لا تسمح هذه الوثيقة بتعديل AAS Runtime المجمد أو فتح تكامل خارجي أو ادعاء قبول تمويلي.

---

## 1. النتيجة التنفيذية

ASIE يملك اليوم طبقة قرار وأدلة أعمق من نموذج دراسة الجدوى التقليدي: محرك مالي، سيناريوهات وحساسية، NPV وIRR وفترة استرداد وDSCR، سجل مخاطر، بوابات جاهزية، Evidence Ledger، شجرة التحويلات، Snapshot غير قابل للتغيير، وحزمة قرار قابلة للمراجعة البشرية.

لكن هذه القوة **ليست بعدُ حزمة تسليم تمويلية مكتملة**. التقرير الحالي هو إسقاط تقني-قراري محفوظ بصيغة HTML؛ لا يطابق بعد التقسيم الرسمي ذي الأقسام الستة عشر، ولا ينتج PDF/Word/Excel تمويلياً، ولا يملك نموذجاً موثقاً لمتطلبات كل جهة أو قطاع.

لذلك القرار الصحيح هو:

```text
مراجع الدراسة الرسمية + مخرجات ASIE العميقة
        ↓
Funder-Ready Report Profile
        ↓
Immutable Snapshot projection
        ↓
PDF/Word/Excel + Evidence Annex + Funding Readiness Checklist
```

لا يجوز أن يعيد قالب التقرير الحساب، أو يملأ نتيجة من الواجهة، أو يعد بقبول أي جهة تمويل.

## 3. مراجعة ASIE الحالية: ما هو أقوى فعلاً

### 3.1 الحقيقة والتدقيق

| قدرة ASIE المثبتة في الكود | القيمة في تقرير تمويلي |
|---|---|
| `snapshot.assemble.v1` يجمع مخرجات وحدات مختومة ويتحقق من الهوية والـhash | يمنع نسبة تقرير إلى تشغيل أو مشروع آخر |
| `build_report()` يربط التقرير بـ Snapshot/Run/Project وcontent/integrity hash | كل تقرير قابل للإرجاع إلى نسخته الأصلية |
| `Decision Pack` يتحقق من تطابق إسقاط التقرير والـoverview قبل إصداره | يمنع تقريراً مفصولاً عن حقيقة القرار |
| Evidence Ledger وTransformation Lineage وQuality Gates | يمكن للمراجع معرفة مصدر أي افتراض وما إذا كان صالحاً لدعمه |
| Human Review overlay منفصل عن Snapshot | توقيع/مراجعة بشرية ممكنان بلا تزوير الحقيقة المختومة |

### 3.2 المالية والقرار

| قدرة حالية | ما تعنيه للممول |
|---|---|
| CAPEX مفصل: معدات، تجهيزات، تراخيص محلية | أساس استخدامات التمويل |
| OPEX مفصل: رواتب، إيجار، مرافق، تسويق، صيانة | أساس التكلفة التشغيلية |
| نموذج طاقة تشغيلية أو وحدات شهرية يدوية | يوضح فرضية حجم النشاط |
| السيناريوهات المتحفظ/الأساسي/المتفائل والحساسيات | يبين مدى هشاشة أو مرونة القرار |
| NPV وIRR وفترة الاسترداد ونقطة التعادل | مقاييس الاستثمار الأساسية |
| خدمة الدين وDSCR مع عتبة ضغط `1.2` | مدخل مباشر لمراجعة القدرة على السداد |
| Monte Carlo مع تحذير صريح بأنه ليس احتمال نجاح غير مشروط | يعرض عدم اليقين من دون ادعاء زائف |
| مخاطر، بوابات جاهزية، وخطة تنفيذ | لا يترك القرار المالي منعزلاً عن قابلية التنفيذ |

### 3.3 الحكم

ASIE ليس مجرد نموذج تعبئة. يستطيع أن يصدر حكماً سيادياً، سبب الحكم، عوائق حرجة، إجراءً تالياً، ومراجعة بشرية منفصلة. هذا أعمق من النموذج المرجعي ومن الواجهات التي تعرض جداول ثابتة فقط.

## 4. الفجوة: ما ينقص قبل تسمية المخرج «جاهزاً للتمويل»

### 4.1 فجوات ليست في AAS Runtime

| الفجوة | حالة اليوم | قرار المعالجة |
|---|---|---|
| هيكل التقرير ذي 16 قسماً | التقرير الحالي يحمل أقسام قرار عامة، لا taxonomy تمويلياً رسمياً | خريطة إسقاط `FRRP` من Snapshot إلى الأقسام 1–16 |
| نموذج مالي متعدد الفترات | المحرك الحالي يحسب تدفقاً سنوياً أساسه ربح شهري متكرر لخمس سنوات | إسقاط مالي تمويلي يضم شهري/ربعي للسنة الأولى وسنوياً للسنوات التالية، بلا تغيير حقيقة المحرك إلا عبر ACR إذا لزم |
| المركز المالي والقوائم الكاملة | توجد CAPEX/OPEX ونتائج قرار؛ لا قالب قوائم تمويلية مكتمل | Contract منفصل للقوائم المالية، مع تحقق المعادلة المحاسبية |
| استخدامات التمويل ومصادره | يوجد funding gap وخدمة دين؛ لا جدول استخدامات/سحوبات/مصادر مفصل | Funding Plan projection مرتبط بـ Snapshot وEvidence |
| متطلبات الجهة/القطاع | لا يوجد كتالوج جهة تمويل أو checklist قابل للمراجعة | Profiles محلية versioned، لا ادعاء قبول شامل |
| حزمة التصدير | HTML موجود؛ لم يثبت مولد PDF/Word/Excel تمويلي في المسار الحالي | Composer بعد اعتماد العقد، مع تحقق parity قبل التصدير |
| ملاحق المستندات | Evidence موجود بنيوياً، لا ترتيب تمويلي للمرفقات | Evidence Annex بفهرس، حالة مراجعة، وروابط/معرفات آمنة |
| اعتماد التقرير | Human Review موجود، لا سياسة اعتماد تمويلي نهائية | Sign-off overlay مع الدور والتاريخ والنطاق، منفصل عن Snapshot |

### 4.2 حدود لا يجوز تجاوزها

- لا تغير React رقماً مالياً أو قراراً أو حالة جاهزية.
- لا يعاد حساب التقرير عند الفتح أو التصدير؛ يقرأ Snapshot محفوظاً فقط.
- لا يفتح اتصال لجهة تمويل أو مصدر حكومي أو AI أو دفع في هذه الحزمة.
- لا يتحول التقرير إلى وعد قبول أو ضمان عائد.
- لا تغير المراجعة البشرية `snapshot_hash` أو الحكم الأساسي؛ تصدر Overlay جديداً قابلاً للتدقيق.
- أي تغيير مطلوب في ترتيب Runtime أو Snapshot Assembly يوقف هذه الخطة ويستلزم ACR منفصلاً.

## 5. تعريف المنتج المقترح: Funder-Ready Report Pack

### 5.1 مخرجات العميل

| المخرج | الغرض | مصدر الحقيقة |
|---|---|---|
| ملخص تنفيذي للتمويل | صفحة قرار واحدة: المشروع، الاحتياج، الاستخدامات، قدرة السداد، القرار والشروط | Snapshot + Decision Pack |
| دراسة جدوى تفصيلية | الأقسام 1–16 بالعربية، والإنجليزية عند طلب Profile ثنائي اللغة | Snapshot projections فقط |
| نموذج مالي | دخل، تدفقات، مركز مالي، تمويل، حساسية، DSCR، رأس مال عامل | Finance projection محفوظ |
| سجل الأدلة والافتراضات | مصدر/تاريخ/جودة/مراجع/تحويل/فجوات | Evidence Ledger + Assumption Book |
| سجل المخاطر وخطة التخفيف | الخطر، المحفز، الأثر، المعالجة، المالك والحالة | Risk Register + review overlay |
| ملحق الجهة التمويلية | قائمة تحقق Profile محدد وما هو مكتمل أو ناقص | Funder Requirement Profile |
| سجل الإصدار | Snapshot ID، hash، تاريخ، المنهجية، المراجع، والمراجعة البشرية | Snapshot Assembly + overlay |

### 5.2 حالات الإخراج

| الحالة | المعنى | ما الذي يسمح به النظام |
|---|---|---|
| `DRAFT_INTERNAL` | المدخلات أو الأدلة ناقصة | معاينة فقط مع العوائق والإجراءات المطلوبة |
| `DECISION_READY` | Snapshot صالح للقرار الداخلي | تقرير قرار داخلي، لا وسم تمويلي |
| `FUNDING_BASE_READY` | الأقسام الأساسية والقوائم والأدلة اجتازت التحقق | حزمة تمويل أساسية قابلة للتصدير |
| `FUNDER_PROFILE_READY` | حزمة الأساس + متطلبات Profile محدد اجتازت التحقق | تقرير يحمل اسم Profile وتاريخ مراجعته |
| `REVIEW_REQUIRED` | تغيرت مدخلات، Profile، أو دليل مؤثر | يمنع استعمال الإصدار السابق كأحدث ملف |

لا توجد حالة اسمها «مقبول لدى جميع الجهات».

## 7. خطة العمل المقترحة

### FR-0 — قرار العقد والحدود

**الهدف:** اعتماد vocabulary وحالات الحزمة ومصفوفة الأقسام ومفهوم الـProfile.  
**المخرجات:** `FunderReportProfile.v1`، `FundingReadinessChecklist.v1`، ومعجم عربي/إنجليزي.  
**قبول المرحلة:** لا تغيير Runtime/Snapshot/API، وتعريف واضح لكل حقل ومن يملكه.  
**Gate:** موافقة المستخدم على العقد قبل أي تنفيذ.

### FR-1 — نموذج البيانات والإسقاطات

**الحالة الحالية:** `IN_PROGRESS — إسقاط v1 منفذ ومختبر`  
**المخرج الحالي:** `backend/funder_report.py` يضيف عقد `funder.report.projection.v1` إلى التقرير المحفوظ، ويعرض الأقسام 1–16، التمويل، سجل الأدلة، والقوائم المالية المتاحة مع فجوات صريحة. لم تُخفَ فجوات المركز المالي أو تفاصيل التدفق الشهري؛ وهي محجوزة لمرحلة استكمال النموذج.

**الهدف:** إكمال فجوات تقرير التمويل من مخرجات ASIE القائمة، لا نسخها إلى نموذج حر.  
**المخرجات:** projections للقوائم، التمويل، الأقسام 1–16، والملاحق.  
**قبول المرحلة:** كل قيمة مالية في التقرير تعود إلى `snapshot_id` و`assumption_ref` أو `evidence_ref`؛ وتفشل الحزمة عند غياب ذلك.  
**قاعدة:** إذا احتاجت المعادلة تغييراً في Runtime، يصدر ACR قبل الكتابة.

### FR-2 — Composer التقرير وقوالب القراءة

**الحالة الحالية:** `IN_PROGRESS — HTML review composer v1 منفذ ومختبر`  
**المخرج الحالي:** مسار معاينة مستقل `GET /api/snapshots/{snapshot_id}/funder-report.html` يعرض النسخة العربية RTL من الإسقاط المحفوظ، مع حالة الجاهزية، الأقسام الستة عشر، التوقعات المتاحة، والفجوات. PDF/Word/Excel لم تُفتح بعد؛ ستأتي بعد تثبيت parity البصري والعقد المالي.

**الهدف:** تحويل الإسقاط إلى تقرير عربي احترافي، ثم قالب إنجليزي مطابق وظيفياً.  
**المخرجات:** HTML review، PDF، Word/Excel attachments عند اعتمادها.  
**قبول المرحلة:** hash/identity parity، RTL سليم، جداول مقروءة، أرقام لا تُعاد حسابها في العرض، ورأس/تذييل يحتويان إصدار Snapshot.

### FR-2b — مخرجات الملفات

**الحالة الحالية:** `IMPLEMENTED — DOCX وXLSX وPDF v1 منفذة ومتحقق منها`  
**المخرجات:** `backend/report_exports.py` يولد DOCX وPDF من الإسقاط نفسه؛ `export_funder_report_pdf` يستخدم محرك PDF خلفياً مثبتاً عبر `ASIE_PDF_RENDERER` أو مسار صريح، مع fallback محلي للتطوير فقط. متصفح العميل (Chrome/Edge/Firefox/Safari) لا يدخل في قرار التصدير؛ يحصل على الملف عبر API. تم التحقق البنيوي والبصري لعينة PDF؛ لا توجد شبكة أو إعادة حساب، ويبقى PDF مخرجاً محلياً يحتاج بوابة الإصدار قبل التسليم الخارجي.

### FR-3 — محرك الجاهزية التمويلية

**الحالة الحالية:** `IN_PROGRESS — funding.readiness.profile.v1 منفذ ومختبر`  
**المخرجات:** ملفات مرجعية محلية قابلة للإصدار (`BASE-FUNDING-V1`، `BANK-SME-BASE-V1`، `DEVELOPMENT-FUND-BASE-V1`، `INVESTOR-EQUITY-BASE-V1`)؛ كل ملف ينتج checks وفجوات وhash مرتبطاً بالـSnapshot. هذه Profiles مرجعية قابلة للتهيئة وليست سياسات معلنة أو ضمان قبول. قائمة الملفات متاحة قراءة عبر `GET /api/funding-profiles`، والجاهزية تظهر داخل `profile_readiness` في الإسقاط.

**الهدف:** منع تسليم تقرير غير مكتمل على أنه ملف تمويل.  
**المخرجات:** checks للأقسام، الأدلة، اتزان المركز المالي، التدفقات، DSCR عند الدين، تفاصيل التمويل، وProfile requirements.  
**قبول المرحلة:** يشرح كل فشل: الدليل/القيمة/القسم الناقص والخطوة التالية، ولا يضع صفراً أو عبارة عامة.

### FR-4 — المراجعة والاعتماد والتسليم

**الهدف:** تمكين مراجع بشري من اعتماد حزمة محددة من دون تعديل الحقيقة.  
**المخرجات:** Review overlay، سجل إصدار، ملحق أدلة، وتقرير فرق بين نسختين.  
**قبول المرحلة:** اعتماد/رفض/طلب تعديل كلها قابلة للتدقيق، والإصدار التاريخي لا يتغير بعد تعديل المشروع.

**الحالة الحالية:** `IN_PROGRESS — funder.report.release.v1 منفذ ومختبر`  
**المخرج الحالي:** `backend/report_release.py` ينشئ سجل إصدار محلياً مربوطاً بـ`projection_hash` و`readiness_hash` وقرار المراجع، مع مسار قراءة `GET /api/snapshots/{snapshot_id}/release`. لا يغيّر السجل Snapshot أو verdict ولا يعلن قبولاً خارجياً.

### FR-5 — Profiles الجهات والقطاعات

**الهدف:** دعم اختلاف المتطلبات من دون ادعاء معيار شامل.  
**المخرجات:** Profile محلي versioned لكل جهة/قطاع تمت مراجعته قانونياً وتشغيلياً.  
**قبول المرحلة:** كل Profile يعلن تاريخ المراجعة، نطاقه، ما لا يغطيه، والمتطلبات التي لا يستطيع ASIE التحقق منها محلياً.

**الحالة الحالية:** `IMPLEMENTED — sector profile catalog v1 منفذ ومختبر`  
**المخرج الحالي:** `backend/funding_readiness.py` يعرّف Profiles مرجعية للتجزئة، الأغذية والمشروبات، التصنيع، والخدمات الرقمية، مع تاريخ مراجعة ونطاق وحدود تحقق صريحة. تعرض عبر `GET /api/sector-profiles`؛ جميعها `reference_only` ولا تمثل اشتراطات جهة رسمية أو ضمان قبول.

## 8. معايير القبول النهائية

1. لا يخرج `FUNDING_BASE_READY` إلا بعد اكتمال جميع الأقسام الإلزامية والأدلة المطلوبة.
2. كل رقم ذي أثر تمويلي قابل للتتبع إلى Snapshot وفرضية/دليل.
3. المركز المالي متزن حسابياً، والتدفقات متوافقة مع قائمة الدخل والتمويل.
4. يوضح التقرير السيناريو الأساسي والمتحفظ والحساسيات؛ لا يعرض رقم عائد منفرداً بلا افتراضاته.
5. يظهر DSCR عند وجود دين، ولا يخفي عدم جاهزية شروط الدين.
6. يميز التقرير بين حقيقة مختومة، افتراض، دليل، مراجعة بشرية، وتوصية.
7. يمكن فتح إصدار قديم ومقارنته بآخر من دون تغيير أي منهما.
8. لا يذكر قبولاً مضموناً أو اعتماداً لجهة تمويل إلا بمرجع تعاقدي/رسمي مراجع خارجياً.
9. لا يخرج ملف خارجي قبل بوابات الأمن والاستعادة والتصدير المعتمدة في المسار التشغيلي.

---

## 9. قرار مطلوب من المستخدم

قبل التنفيذ، يعتمد المستخدم أحد الخيارين:

1. **اعتماد FR-0 فقط:** توثيق العقود والقوالب وقائمة الجاهزية دون أي تعديل كود.
2. **اعتماد FR-0 إلى FR-4:** بدء تنفيذ حزمة تقرير تمويلي محلية بعد مراجعة كل مرحلة وGate مستقل.

حتى ذلك الحين، تبقى هذه الوثيقة مراجعة وخطة عمل فقط.

## 10. أدلة المراجعة الحالية

- `backend/reports.py`: يبني تقريراً من Snapshot ويعرض المالية، الأدلة، المخاطر، التنفيذ والجاهزية؛ والمخرج HTML حالياً.
- `backend/decision_pack.py`: يتحقق من هوية Snapshot وhashes ويسجل المراجعة كـoverlay منفصل.
- `backend/finance_engine.py`: يوفّر CAPEX/OPEX، التمويل، DSCR، NPV، IRR، نقطة التعادل، السيناريوهات والحساسيات.
- `backend/snapshot_assembly.py`: يحصر assembly في envelopes مختومة وعقد `snapshot.assemble.v1`.
- `tests/test_local_platform.py`: يتضمن اختبارات ثبات Snapshot، parity التقرير/Decision Pack، ومراجعة الأدلة والتحويلات.
- `نموذج دراسة الجدوى (1).pdf` و`Feasibility Study Model.pdf`: النموذجان المرجعيان المرفقان في جذر مساحة العمل.

**حالة التحقق:** هذه مراجعة قراءة للكود والوثائق والنماذج. لم تُشغّل اختبارات أو build جديد ضمن هذه الوثيقة؛ أي نتائج نجاح سابقة تبقى دليلاً مسجلاً وليست تحققاً جديداً.
