# ASIE — عقد التقرير التمويلي وقائمة الجاهزية

**الإصدار:** `FunderReportProfile.v1` / `FundingReadinessChecklist.v1`  
**الحالة:** `FR-0 DRAFT — يحتاج اعتماد Gate`  
**المرجع:** `ASIE-FRRP-01-2026-07-20`

## 1. الغرض والحدود

هذا العقد يعرّف شكل حزمة التقرير التي يستلمها العميل أو يرفعها للممول. وهو عقد إسقاط وعرض فوق الحقيقة الموجودة في `Snapshot`؛ لا يملك الحسابات ولا يغيّر `AAS Runtime` أو `snapshot.assemble.v1`.

القواعد غير القابلة للتفاوض:

- كل رقم أو حكم أو ادعاء يحمل `snapshot_id`، ومعرّف افتراض أو دليل عند انطباقه.
- لا يعاد الحساب عند فتح التقرير أو تصديره.
- المراجعة البشرية طبقة منفصلة ولا تعدّل Snapshot أو verdict الأصلي.
- لا يوجد ادعاء قبول عام؛ الجاهزية دائماً مرتبطة بملف جهة/قطاع محدد.
- أي تغيير في Runtime أو Assembly يتطلب `Architectural Change Request` مستقلاً.

| سجل الإصدار | Snapshot ID، hashes، المنهجية، المراجع، الاعتماد | Snapshot Assembly + overlay |

## 3. قاموس حالات الحزمة

| الحالة | المعنى | الإجراء المسموح |
|---|---|---|
| `DRAFT_INTERNAL` | بيانات أو أدلة ناقصة | معاينة داخلية فقط |
| `DECISION_READY` | Snapshot صالح لقرار داخلي | تقرير قرار داخلي |
| `FUNDING_BASE_READY` | الأقسام والقوائم والأدلة الأساسية اجتازت الفحص | تصدير حزمة تمويل أساسية |
| `FUNDER_PROFILE_READY` | الحزمة اجتازت متطلبات Profile محدد | تصدير باسم الجهة/القطاع وتاريخ المراجعة |
| `REVIEW_REQUIRED` | تغير دليل أو افتراض أو Profile مؤثر | إيقاف الإصدار السابق وإعادة المراجعة |

لا تُستخدم كلمات مثل «مقبول»، «مضمون»، أو «يقبل لدى جميع الجهات» كحالة نظامية.

## 5. FundingReadinessChecklist.v1

### أ. الهوية والنطاق

- [ ] اسم المشروع والكيان والموقع والقطاع والمرحلة مكتملة.
- [ ] `snapshot_id` ونسخة المنهجية ووقت التوليد مثبتة.
- [ ] لا توجد مدخلات غير معرّفة أو تغييرات بعد Snapshot.

### ب. الأدلة والافتراضات

- [ ] كل ادعاء مؤثر له `evidence_ref` أو `assumption_ref`.
- [ ] الأدلة اجتازت Quality Gate أو ظهرت فجوتها صراحة.
- [ ] سجل التحويلات والمصادر والتواريخ قابل للمراجعة.

### ج. السوق والتشغيل

- [ ] السوق والمنافسون والطلب والقطاعات المستهدفة موثقة.
- [ ] خطة التسويق والتشغيل والموارد البشرية والجدول الزمني مترابطة.
- [ ] المخاطر الحرجة لها تخفيف ومالك وموعد.

### د. المالية والتمويل

- [ ] CAPEX وOPEX والطاقة التشغيلية موثقة.
- [ ] قائمة الدخل والتدفقات والمركز المالي متسقة؛ المركز المالي يحقق `الأصول = الخصوم + حقوق الملكية`.
- [ ] نقطة التعادل وNPV وIRR وفترة الاسترداد وDSCR والحساسية متسقة مع Snapshot.
- [ ] مصادر التمويل واستخداماته وتوقيته وفجوة التمويل واضحة.

### هـ. التصدير والاعتماد

- [ ] اجتياز parity بين التقرير وDecision Pack وSnapshot hashes.
- [ ] RTL، الجداول، الأرقام، الرأس والتذييل صالحة للقراءة والطباعة.
- [ ] Profile الجهة/القطاع محدد بإصدار وتاريخ، أو وسم التقرير `FUNDING_BASE_READY` فقط.
- [ ] المراجع البشرية والأدوار والنطاق والتاريخ مسجلة كـoverlay.

## 6. معايير Gate FR-0

تُعد FR-0 مكتملة عندما:

1. يعتمد المستخدم هذا العقد وقاموس الحالات.
2. يوافق على أن التقرير إسقاط Snapshot لا محرك حساب مستقل.
3. يوافق على فصل الجاهزية الأساسية عن متطلبات كل جهة أو قطاع.
4. يوافق على عدم إجراء أي تعديل برمجي قبل اعتماد العقد.

**قرار الاعتماد:** `PENDING_USER_APPROVAL`

بعد الاعتماد فقط يبدأ FR-1، مع مراجعة مستقلة قبل كل مرحلة لاحقة.
